# Сборка, релизы и деплой

Этот документ описывает, как собирать и деплоить платформу AniCore.

## Сборка локально

Основные команды сборки выполняются из репозитория AniCore (корень `anidev/`):

```bash
cd /home/USER/anidev

# Сборка всех компонентов
make build

# Отдельные цели
make build-runtime    # runtime (бинари/артефакты из AniRuntime)
make build-cores      # core-модули (.so)
make build-plugins    # плагины (.so)
```

Перед сборкой убедитесь, что:

- настроен `go.work` (см. `getting-started.md`),
- переменная `CGO_ENABLED` установлена в `1`:

```bash
export CGO_ENABLED=1
```

## Артефакты сборки

Обычно структура артефактов выглядит так (может отличаться в зависимости от Makefile):

```text
build/
├── runtime/           # бинарь runtime
├── cores/             # core-модули (.so)
└── plugins/           # плагины (.so)
```

Пути к core-модулям и плагинам должны соответствовать значениям в `config.json` (поля `core_directory`, `plugins.directory`).

## Конфигурация для продакшена

Рекомендуется иметь отдельный `config.prod.json` с боевыми настройками:

- реальные токены и ключи брать из переменных окружения или секрет-хранилища,
- логи отправлять в централизованную систему (через настройки `pkg/logger` или внешний wrapper),
- включать/выключать backend, нужные плагины и core-модули.

При запуске runtime можно указать нужный конфиг:

```bash
./anicore-runtime -config config.prod.json
```

## CI/CD

Общая схема:

1. **AniApi**: при изменении публичного API:
   - прогонить `go test`, `go vet`, линтеры,
   - собрать модуль и опубликовать тег/релиз (опционально).

2. **AniCore**: при изменениях в ядре/core-модулях/плагинах:
   - собрать core-модули и плагины,
   - прогнать тесты и линтеры,
   - при необходимости собрать Docker-образ backend/core.

3. **AniRuntime**:
   - собрать бинарь runtime для целевой платформы,
   - прогнать тесты,
   - при необходимости упаковать в Docker.

4. **AniDocs**:
   - прогнать проверки форматирования/линтер для Markdown (если настроены),
   - опубликовать документацию (например, на GitHub Pages или в другом хостинге).

## Обновление документации как submodule

Если AniDocs подключён как submodule в AniCore и AniRuntime:

```bash
# Обновить документацию до последней версии
cd /home/USER/anidev
git submodule update --remote docs

git add docs
git commit -m "docs: обновлена версия AniDocs"
git push
```

## Деплой

Возможные варианты деплоя:

- **Standalone бинарь**:
  - собрать runtime,
  - скопировать `config.json`, `.so` core-модулей и плагинов,
  - запустить на целевом сервере под systemd/supervisor.

- **Docker**:
  - собрать Docker-образ, который содержит:
    - бинарь runtime,
    - core-модули и плагины,
    - конфиг (или путь к нему через volume),
  - деплоить через Docker Compose/Kubernetes.

- **Backend**:
  - backend core поднимает HTTP API и dashboard,
  - может деплоиться как часть того же процесса runtime или как отдельный сервис, в зависимости от архитектуры.

## Проверка после деплоя

1. Проверить логи runtime — должны быть сообщения о загрузке всех core-модулей и плагинов без ошибок.
2. Убедиться, что backend (если включён) доступен по HTTP.
3. Для Discord core — проверить, что бот онлайн и slash-команды зарегистрированы.
4. Прогнать базовый sanity-check (например, команда `/ping` или другой простой плагин).

Подробнее о конфигурации и архитектуре см. в `configuration.md` и `architecture.md`.
