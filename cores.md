# Разработка core-модулей

Core-модули — это компоненты, которые подключают AniCore к внешним системам: Discord, Telegram, backend-сервер, и т.д.

Они:

- собираются как `.so` файлы,
- загружаются runtime через `coreloader.Loader`,
- получают доступ к runtime через интерфейс `RuntimeAPI` (определён в `AniApi/shared`).

## Основная идея

Core-модуль:

- реализует интерфейс `shared.Core` (см. `AniApi/shared/types.go`),
- получает `RuntimeAPI` в методе `Init`,
- запускает свои фоновые процессы в `Start`,
- корректно завершает их в `Stop`,
- генерирует события `shared.Event` и публикует их через `RuntimeAPI.Emit`.

## Взаимодействие с Runtime

`AniRuntime` при загрузке core-модуля делает следующее:

1. Загружает `.so` через `coreloader.Loader`.
2. Получает инстанс core-модуля, реализующий `shared.Core`.
3. Формирует конфигурацию для core-модуля из `config.CoreConfigs[coreName]`.
4. Создаёт `RuntimeAPIImpl` с ссылкой на `Runtime` и конфиг.
5. Вызывает `core.Init(runtimeAPI)`.
6. Позже вызывает `core.Start()` и при завершении — `core.Stop()`.

## RuntimeAPI

Интерфейс `RuntimeAPI` (см. `AniApi/shared`) предоставляет core-модулям доступ к:

- логированию (`Log`),
- публикации событий (`Emit`),
- регистрации/получению других core-модулей,
- получению конфигурации core-модуля (`GetConfig`),
- получению списка команд плагинов (`GetPluginCommands`),
- получению версии ядра (`GetCoreVersion`),
- доступу к базе данных (`GetDatabase`).

Конкретная реализация в runtime — `RuntimeAPIImpl`:

- хранит ссылку на `Runtime`, core-модуль и его конфиг,
- добавляет префикс `Core:<name>` к логам,
- предоставляет метод `GetPluginCommands()`, который собирает все команды плагинов.

## Пример сценария для Discord core

Типичный Discord core-модуль будет делать:

1. В `Init` получить токен и конфигурацию из `RuntimeAPI.GetConfig()`.
2. Создать Discord-сессию (`discordgo.New("Bot " + token)`).
3. В `Start` подключиться к Discord, зарегистрировать slash-команды:
   - вызвать `api.GetPluginCommands()` из `RuntimeAPI`,
   - зарегистрировать их через Discord API.
4. Подписаться на события Discord (сообщения, интеракции),
5. Преобразовывать их в `shared.Event` и публиковать через `RuntimeAPI.Emit(e)`.

## Backend core

Backend core-модуль реализует HTTP API и dashboard:

- в `Init` получает конфиг (порты, пути, настройки БД),
- в `Start` поднимает HTTP-сервер,
- использует БД через `RuntimeAPI.GetDatabase()`,
- может публиковать события при изменении данных.

## Конфигурация core-модуля

Конфигурация передаётся как `map[string]interface{}` и зависит от конкретного модуля.

Примеры полей для Discord core:

- `token` — токен бота,
- `guild_id` — ID сервера для регистрации команд,
- дополнительные настройки (префиксы, режим debug и т.д.).

Конкретный формат описывается в документации самого core-модуля и в `configuration.md`.

## Сборка core-модулей

Core-модули, как и плагины, собираются в `.so` файлы.

Обычно есть `Makefile` с таргетом `make build-cores`, который:

- собирает каждый core-модуль в `./build/cores/<name>.so`,
- использует `CGO_ENABLED=1`.

Runtime затем берёт имена из `config.Cores` и ищет соответствующие файлы `.so` в директории `core_directory`.

## Логирование и ошибки

Правила те же, что для остальной части проекта:

- использовать `RuntimeAPI.Log` (внутри — `pkg/logger`),
- не использовать стандартный `log`,
- оборачивать ошибки через `fmt.Errorf` и `%w`,
- сообщения об ошибках начинать со строчной буквы.

## Взаимодействие с плагинами

Core-модуль не обращается к плагинам напрямую, а работает через события:

- события от внешних систем → `shared.Event` → `RuntimeAPI.Emit(e)` → плагинам через `OnEvent(e)`,
- ответы плагинов могут генерировать новые события, которые обрабатывает core-модуль.

Для команд Discord core получает список команд через `RuntimeAPI.GetPluginCommands()` и регистрирует их в Discord.

Подробнее об общих типах и интерфейсах см. в `api.md` и `architecture.md`.
